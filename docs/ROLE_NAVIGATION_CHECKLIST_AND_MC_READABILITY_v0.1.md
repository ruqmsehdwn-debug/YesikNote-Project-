# YesikNote 역할별 탐색·체크표·MC 가독성 v0.1

> 상태: PROPOSED
>
> 범위: W5 Role Navigation·Checklist·MC Readability
>
> 정책 확정: Product Owner 검토 전까지 확정 정책으로 사용하지 않음

## 1. 목적

Owner가 13~15개 식순을 빠르게 찾고, 예식장 체크표는 핵심 정보를 훑어보기 쉬우며, 사회자는 실제 대본을 멀리서도 안정적으로 읽을 수 있게 한다.

이번 변경은 기존 Draft 데이터와 Script Engine의 결과를 바꾸지 않는다. 기존 데이터를 역할별 화면에서 더 분명하게 탐색하고 읽도록 표현만 개선한다.

## 2. 식순 탐색 패턴

### 선택한 방식

- 데스크톱: 왼쪽 고정 목록
- 태블릿·모바일: 상태가 포함된 native select와 현재 선택 요약
- 공통 라벨: `식순 선택`

`식순 선택`은 현재 단계에서 조회와 편집 이동을 함께 설명하면서도 `편집할 식순`보다 짧고 직접적이다.

### 데스크톱

목록은 순번, 식순명, `작성 완료`·`확인 필요`·`미진행` 상태를 함께 보여준다. 선택 항목은 색상과 `aria-current="step"`으로 표시한다. 위·아래 방향키와 Home·End 키로 이동할 수 있다.

### 태블릿·모바일

작은 화면에서는 긴 목록이 편집 영역을 밀어내지 않도록 native select를 유지했다. 각 option에 식순명과 상태를 함께 표시하고, 선택 결과를 아래에 다시 보여준다. 선택 컨트롤의 최소 높이는 44px이다.

### 데이터 안전성

- 식순 선택은 안정 ID를 사용한다.
- 순서가 바뀌어도 배열 index가 아닌 ID로 같은 항목을 찾는다.
- `active = false` 항목도 `미진행` 상태로 목록에 남는다.
- 탐색은 입력값, Draft, 완료 상태를 수정하지 않는다.

## 3. 주례 조건부 표시

### 주례 없는 예식

- `예식 형태: 주례 없는 예식`만 표시한다.
- 같은 의미의 `주례 유무: 무주례` 카드는 표시하지 않는다.
- 기존 무주례 Canonical 식순과 사용자가 저장한 순서는 변경하지 않는다.

### 주례 있는 예식

- `예식 형태: 주례 있는 예식`을 표시한다.
- 기존 `officiant_entrance` 참여자 데이터에서 주례 성함과 `직함 또는 관계`를 읽어 `주례 정보`로 표시한다.
- 이름이나 관계가 없으면 임의로 채우지 않고 `확인 필요`와 warning으로 남긴다.

### Canonical 적용 기준

현재 코드와 자동 테스트가 보장하는 주례 없음 13개, 주례 있음 15개 순서를 유지한다. 이번 변경은 Canonical 생성 로직, 예식 유형 변경 흐름, 기존 Draft의 사용자 순서를 수정하지 않는다.

## 4. 예식장 체크표 타이포그래피

| 영역 | 적용 기준 |
|---|---|
| 요약 카드 라벨 | 14px, 700 |
| 요약 카드 값 | 17px, 600 |
| 전체 표 식순명 | 17px, 700 이상 |
| 전체 표 본문 | 14px |
| 전체 표 Cue·Note | 13px, 보조 색상과 충분한 행간 |
| A4 식순명 | 11.5pt, 700 |
| A4 본문 | 9.5pt |
| A4 Cue·Note | 9.25pt |

긴 문장은 셀 안에서 줄바꿈한다. 식순명은 본문보다 크고 굵게 표시해 화면과 인쇄물 모두에서 행의 시작점을 빠르게 찾을 수 있게 한다.

## 5. MC 폰트와 가독성

### 폰트

MC 본문은 다음 순서의 산세리프 글꼴을 사용한다.

`Pretendard, -apple-system, system-ui, "Apple SD Gothic Neo", "Noto Sans KR", sans-serif`

저장소에 폰트 바이너리, CDN 로더, 새 package를 추가하지 않았다. 따라서 이번 변경으로 새 글꼴 파일을 재배포하거나 별도 라이선스 파일을 추가하지 않는다. Pretendard가 기기에 있으면 우선 사용하고, 없으면 운영체제의 한국어 산세리프로 안전하게 대체한다. 모든 환경에서 Pretendard를 강제로 제공하는 방식은 폰트 자산·출처·라이선스 검토와 별도 승인이 필요한 후속 결정이다.

### 위계

| 정보 | 데스크톱 | 태블릿 | 모바일 |
|---|---:|---:|---:|
| 현재 식순명 | 22px | 반응형 유지 | 반응형 유지 |
| 실제 대본 | 28px | 26px | 22px |
| 대본 굵기·행간 | 550 · 1.72 | 550 · 1.72 | 550 · 1.7 |
| Cue | 16px · 600 | 동일 | 동일 |
| Note | 15px · 450 | 동일 | 동일 |
| 다음 식순 | 16px | 동일 | 동일 |

대본 한 줄은 최대 약 36em으로 제한한다. Cue는 짧은 현장 행동으로, Note는 그보다 약한 실행 메모로 구분한다. 기존 직접 수정 대본과 Script Engine 결과는 변경하지 않는다.

## 6. 전체 화면 감사 결과

| 화면 | 발견한 문제 | 수정 내용 | 남은 문제 |
|---|---|---|---|
| Owner 상세 설정 | 13~15개 식순을 select만으로 오가기 느림 | 데스크톱 고정 목록, 상태 배지, 키보드 이동 추가 | 작은 화면은 OS별 native select 표현 차이가 있음 |
| 태블릿·모바일 Owner | 현재 항목의 상태가 select 밖에서 보이지 않음 | 현재 선택과 상태 요약, 44px 터치 높이 추가 | 별도 sheet 탐색은 이번 범위에서 제외 |
| 예식장 요약 | 예식 형태와 주례 유무가 중복됨 | 주례 유무 카드 제거, 주례 예식만 주례 정보 조건부 표시 | 주례 데이터가 비어 있으면 현장 확인 필요 |
| 최종 전체 표 | 식순명과 일반 문장의 위계가 약함 | 식순명·본문·Cue·Note 크기와 굵기 분리 | 매우 긴 현장 메모는 인쇄 페이지가 늘어날 수 있음 |
| A4 인쇄 | 식순명과 본문이 비슷하게 보임 | 식순명 11.5pt/700, 본문과 Cue·Note 분리 | 실제 OS 인쇄 대화상자별 여백 차이는 별도 실물 확인 필요 |
| MC 화면 | 명조 계열 본문과 작은 글자로 긴 낭독이 어려움 | 한국어 산세리프 우선, 반응형 크기·행간·줄 길이 적용 | 전용 외부 폰트 배포는 별도 승인 필요 |

## 7. 테스트와 실제 QA

- 자동 테스트: 13개 파일, 183개 테스트 통과
- build: 통과
- TypeScript typecheck: 통과
- `git diff --check`: 통과
- 데스크톱 1280px와 1440px: 목록 탐색, 선택 강조, 상태 표시, 키보드 이동 확인
- 태블릿 768px: select 전환, 단일 열 편집, 가로 넘침 없음
- 모바일 390px: select 44px, 현재 상태 표시, Owner·MC 가로 넘침 없음
- MC: 데스크톱 28px, 태블릿 26px, 모바일 22px 대본과 Cue·Note 위계 확인
- 최종 체크표: 중복 `주례 유무` 없음, 카드 라벨 14px, 값 17px, 전체 표 식순명 17px 확인
- 브라우저 Console: 확인한 화면에서 오류 없음
- A4: 인쇄 CSS와 자동 테스트로 위계를 확인함. 실제 OS 인쇄 대화상자의 종이 출력은 MISSING EVIDENCE이다.
- 주례 있는 예식: 자동 테스트로 주례 정보와 Canonical을 확인함. 기존 사용자 Draft를 보호하기 위해 실제 저장 Draft의 예식 유형은 변경하지 않았다.

## 8. 변경하지 않은 영역

- Draft key `yesiknote:owner-builder:draft:v1`
- Draft schema `2`
- localStorage 읽기·쓰기·migration
- Autosave와 한국어 IME 처리
- Script Engine 핵심 로직과 승인 대본
- 직접 수정 대본
- 공연 카드와 안정 ID
- package, dependency, Vite·배포 설정
- DB, Supabase, Auth, RLS

## 9. 미확정 사항

- 모바일 전용 sheet 형태 탐색이 native select보다 유리한지 파일럿에서 확인이 필요하다.
- Pretendard를 모든 기기에 동일하게 제공할지는 자산 배포와 라이선스 검토 후 결정해야 한다.
- 실제 예식장 프린터와 OS 인쇄 대화상자를 이용한 A4 실물 검증이 필요하다.
- 주례의 직함과 관계를 장기 데이터 모델에서 분리할지는 별도 데이터 계약 결정이 필요하다.

## 10. 개발자 전달 요약

1. 신랑·신부는 데스크톱에서 식순 목록을 보고 원하는 항목으로 바로 이동할 수 있다.
2. 모바일에서는 큰 선택 상자와 현재 상태를 함께 보여준다.
3. 이동은 안정 ID만 사용하므로 식순 순서가 바뀌어도 다른 항목을 잘못 열지 않는다.
4. 미진행 식순도 삭제하지 않고 목록에 남긴다.
5. 무주례 예식의 중복 정보는 제거했다.
6. 주례 예식은 기존 참여자 데이터로 주례 이름과 관계를 보여준다.
7. 예식장 체크표는 제목, 값, Cue, Note가 서로 다른 크기로 보인다.
8. MC 대본은 큰 산세리프 글자와 넓은 행간으로 읽기 쉽게 했다.
9. Draft, Autosave, Script Engine, 직접 수정 대본은 건드리지 않았다.
10. 실제 기능 근거는 자동 테스트와 390·768·1280·1440px 화면 QA로 확인했다.
