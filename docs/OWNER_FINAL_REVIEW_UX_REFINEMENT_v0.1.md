# YesikNote Owner 최종 확인 UX 개선 v0.1

상태: **PROPOSED · 로컬 구현 및 자동 검증 완료 · Release 전**

## 1. 문서 목적

Owner가 축가·축무 정보를 처음 입력하는 흐름을 단순화하고, 필수 작성 완료와 현장 확인 필요를 서로 다른 상태로 이해할 수 있게 한다. 또한 최종 식순의 확인 필요 항목에서 기존 안정 ID를 사용해 정확한 입력 위치로 돌아가는 규칙을 기록한다.

이 문서는 W4.1 로컬 구현의 근거다. Commit, Push, PR, merge, 배포 또는 Product Owner 정책 확정을 의미하지 않는다.

## 2. 축가·축무 공연 입력 UX

### IMPLEMENTED

- `performance` 식순이 `active=false`에서 `active=true`로 바뀌는 순간 공연 카드가 0개이면 카드 1개를 생성한다.
- 생성되는 카드는 기존 `PerformanceItem` 구조와 `createId('performance')` 안정 ID를 사용한다.
- 첫 카드를 만든 뒤 상세 설정 단계, 해당 카드, `축가자` 입력칸으로 이동한다.
- 이미 카드가 있으면 새 카드를 만들지 않는다.
- 상세 화면에서 사용자가 마지막 카드를 삭제하면 즉시 다시 생성하지 않는다.
- 빈 상태에는 `등록된 공연이 없습니다`, 안내 문장, `첫 공연 추가` 버튼을 표시한다.
- 식순의 `active` 상태는 카드 삭제 때문에 바꾸지 않는다.
- `Draft` key와 schema는 변경하지 않는다.

### 공연자 라벨

| 공연 종류 | 입력 라벨 |
|---|---|
| 축가 | 축가자 |
| 축무 | 공연자 |
| 축주 | 연주자 |

관계 문구의 읽기 좋은 표현은 Projection 표시 단계에서만 적용한다. 예를 들어 `신랑의 고등학교 친구`는 `신랑 고등학교 동창`으로 표시할 수 있지만 저장된 원문은 바꾸지 않는다.

## 3. 공연 카드 자동 생성 정책

자동 생성 조건은 다음을 모두 만족할 때다.

1. 대상 식순의 type이 `performance`다.
2. 사용자가 미진행 식순을 진행으로 활성화했다.
3. 기존 공연 카드가 0개다.

렌더링, Autosave, Projection 생성만으로 카드를 만들지 않는다. 따라서 마지막 카드를 직접 삭제한 직후에도 사용자의 삭제 결정을 덮어쓰지 않는다.

## 4. 공연 카드 삭제 후 빈 상태

마지막 카드 삭제는 배열을 빈 배열로 보존한다. 화면은 다음 행동을 명확히 제시한다.

- 제목: `등록된 공연이 없습니다`
- 설명: `축가·축무를 진행하려면 공연 정보를 추가해 주세요.`
- 버튼: `첫 공연 추가`

활성 공연 식순에 카드가 없으면 기존 Validation에서 확인 필요 항목으로 유지된다.

## 5. 작성 완료와 현장 확인 상태

### IMPLEMENTED

최종 확인 상단은 다음 상태를 분리한다.

| 구분 | 표시 기준 |
|---|---|
| 작성 상태 | blocking Validation이 없으면 `필수 입력 완료` |
| 수정 필요 | Owner가 직접 고칠 blocking Validation만 `USER_ACTION N개` |
| 현장 확인 | 예식장과 확인할 Projection 정보만 `FIELD_CONFIRMATION N개` |
| 내부 정책 | `POLICY_INTERNAL`로 분리하고 Owner 개수·경고에서 제외 |
| 최종 확인 | USER_ACTION과 FIELD_CONFIRMATION이 모두 0개일 때만 `최종 확인 완료` |

`필수 입력 완료`와 `현장 확인 N개`는 동시에 존재할 수 있다. USER_ACTION만 빨간 수정 영역에 표시하고 FIELD_CONFIRMATION은 파란색 중립 영역에 표시한다.

## 6. 확인 필요 클릭 이동 규칙

### IMPLEMENTED

- 기본정보 오류는 해당 `data-basic-field` 입력으로 이동한다.
- 일반 식순 오류는 `ceremonyItemId` 또는 `itemId`로 식순을 찾고 `data-item-field` 입력으로 이동한다.
- 공연 오류는 `ceremonyItemId`와 `performanceId`를 함께 사용한다.
- 순서 번호와 배열 index는 이동 기준으로 사용하지 않는다.
- 카드와 입력칸을 스크롤한 뒤 잠시 강조하고 입력칸에 focus한다.
- 이동 완료 토스트는 `확인할 입력칸으로 이동했습니다.`다.
- 입력 필드가 아직 없는 내부 정책 항목은 Owner 화면에 표시하지 않는다.
- 이동 과정은 기존 입력값을 변경하지 않는다.
- `한 번에 확인하기`는 USER_ACTION만 대상으로 현재 React 세션에서 큐를 유지한다.
- 각 항목을 해결하면 `다음 확인 항목`으로 이어지고 마지막 항목 뒤 최종 확인으로 돌아온다.
- FIELD_CONFIRMATION과 POLICY_INTERNAL은 연속 수정 흐름에 포함하지 않는다.

이 규칙은 상단 확인 요약, 최종 식순의 상태 배지·`바로 수정`, 상세 확인 문장, 예식장 확인 필요 항목에서 재사용한다.

## 7. 식순명 시각적 위계

### IMPLEMENTED

- 식순명은 데스크톱 기준 20px, font-weight 700으로 표시한다.
- 진행 요약은 13px 보조 문장으로 분리한다.
- 순번은 작은 번호 배지로 표시한다.
- 상태 배지는 식순명보다 작은 10px 보조 요소다.
- `자세히 보기`와 `바로 수정`을 별도 동작으로 구분한다.
- 주요 행동의 최소 높이는 44px로 유지한다.
- 700px 이하에서는 목록의 제목 행과 상태·행동 행을 분리하는 반응형 grid를 적용한다.

## 8. 상세 문장 포맷

### IMPLEMENTED

- 혼인서약은 진행 여부와 낭독자를 별도 행으로 표시한다.
- 성혼선언은 진행자와 관계를 별도 행으로 표시한다.
- 신부 입장은 등장 방식과 입장 방식을 별도 행으로 표시한다.
- 축가 카드는 Product Owner 확정 정책에 따라 카드 1개를 1곡으로 표시한다.
- 축무·축주·기타 공연은 `공연 N건`으로 표시할 수 있다.
- 공연 종류, 곡명·공연명, 축가자·공연자·연주자, 관계를 별도 행으로 표시한다.
- Cue와 Note는 서로 다른 영역으로 유지한다.
- 기본 Cue는 표시 전용 파생 함수에서 핵심 1~2개로 줄인다.
- 사용자가 직접 입력한 Cue는 줄이거나 삭제하지 않는다.

## 9. 역할별 화면 책임

| 역할 | 기본 강조 정보 | 기본 접힘 정보 |
|---|---|---|
| Owner | 식순명, 진행 주체, 확인 필요, 바로 수정 | 예식장 전체 요약, 현장 비고, 전체 표, 사회자 전체 대본 |
| 예식장·인쇄 | 전체 식순, 진행자, Cue, Note, 준비·확인 상태 | 해당 없음 |
| 사회자 | 현재 대본, 다음 식순, Cue, Note | Owner 편집 상세 |

세 역할은 같은 Ceremony Data와 Projection을 사용하지만, 화면에서 먼저 보여주는 정보는 다르다.

## 13. 개발자 전달 요약

1. 공연 식순을 처음 켤 때 카드가 없으면 카드 1개를 만든다.
2. 이미 카드가 있으면 중복 생성하지 않는다.
3. 마지막 카드를 사용자가 지우면 다시 만들지 않고 빈 상태를 보여준다.
4. Owner가 고칠 항목과 예식장에서 확인할 항목을 빨간색·파란색으로 나눈다.
5. 내부 정책은 Owner 경고 수에서 제외한다.
6. 오류 이동은 화면 번호가 아니라 식순·공연의 안정 ID를 사용한다.
7. 여러 오류는 다음 확인 항목으로 연속 이동한다.
8. 축가 카드 1개는 축가 1곡으로 표시한다.
9. 기본 Cue는 짧게 보이되 사용자가 입력한 Cue는 그대로 보존한다.
10. Draft key/schema와 Script Engine 핵심 로직은 변경하지 않았다.
