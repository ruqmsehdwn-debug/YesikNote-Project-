# YesikNote 예식장 체크표 데이터 매핑 v1.0

> 문서 상태: `PROPOSED` — Product Owner 검토 전 구현 계약으로 확정하지 않음
>
> W2.5 로컬 QA 상태: Projection 순수 함수와 Owner 최종 확인 단계의 읽기 전용 Preview는 `IMPLEMENTED`이며 자동 테스트·build·로컬 Owner 화면 QA로 `VERIFIED`했다. 아직 Commit·배포·Product Owner 확정 전이다.
>
> Batch: W2 — 개별 예식 샘플 식순표 기준 정리
>
> 범위: 자료·코드 읽기와 데이터 연결 명세만 수행. 제품 코드·Draft·schema는 변경하지 않음

## 1. 문서 목적

신랑·신부가 한 번 입력한 정보를 다음 두 흐름에서 같은 의미로 사용하는 기준을 제안한다.

- **A. 한 예식 기준:** Owner 입력 → 개별 예식 최종 식순표 → MC 대본·Cue·Note
- **B. 여러 예식 운영 기준:** 각 개별 예식 최종 식순표 → 날짜·시간별 통합 운영 체크리스트

두 결과물은 목적과 모양이 다르다. 개별 최종 식순표는 한 예식의 공동 미팅·현장 실행 기준이고, 통합 운영 체크리스트는 여러 예식의 자료 준비와 확인 상태를 빠르게 점검하는 운영표다. 같은 양식으로 합치거나 수동 복사본으로 따로 관리하지 않는다.

두 결과물은 파일럿 운영에 중요한 목표다. 다만 현재 W2-Finalize 지침에 따라 개별 최종 식순표는 **파일럿 전 최소 기능 후보**, 주말 통합 운영표는 여러 예식 저장·권한·서버 저장 선행 조건을 함께 검토할 대상으로 기록한다. 정확한 구현 시점은 Product Owner 결정 전 확정하지 않는다. 이 문서는 설계 제안이며 Final Snapshot, 서버 저장, 인증, 역할 권한, 인쇄·PDF를 구현하거나 승인하지 않는다.

## 2. 근거 자료와 확인 수준

개인정보 보호를 위해 원본의 이름·날짜·예식장 정보와 예시 메모는 이 문서에 옮기지 않았다.

| 근거 | 역할 | 확인 수준 | 실제 확인 내용 | 한계 |
|---|---|---|---|---|
| 개별 예식 샘플 `.pages` | 한 예식의 최종 식순표 | `VERIFIED` | 실제 파일과 1쪽 A4 표를 열어 3개 열, 12개 식순 행, `축가` 비고의 `1곡`을 확인 | 빈 셀의 의도는 문서만으로 알 수 없음 |
| 주말 운영 체크리스트 `.numbers` | 여러 예식의 날짜·시간별 운영표 | `PARTIAL` | 실제 패키지의 전체 미리보기에서 2일 구획, 여러 시간대, 반복 점검 행과 상태·메모 셀 구조를 확인 | 원본의 작은 셀 문구 전체와 숨은 시트·인쇄 범위는 정확히 판독하지 못함 |
| 주말 운영 체크리스트 PDF | `.numbers`와 같은 내용인지 확인할 PDF | `MISSING EVIDENCE` | 첨부된 실제 파일은 `.numbers`이며 별도 PDF는 제공되지 않음 | PDF 전체 페이지·인쇄 결과를 읽었다고 판단할 수 없음 |
| 현재 저장소 코드 | 현재 MVP 데이터·대본 흐름 | `VERIFIED` | `CeremonyDraft`, Canonical 템플릿, Script Engine, Draft 저장, Owner·MC 화면을 읽기 전용 확인 | 실행 테스트는 이번 Batch 금지 범위라 수행하지 않음 |
| 저장소 기준 문서 | 승인된 고정 계약과 미확정 범위 | `VERIFIED` | Canonical 13/15, 화촉 A8/B7, Draft key/schema 2, 안정 ID, `active=false`, Script Engine 공유 기준 확인 | 새 Product Owner 결정과 기존 파일럿 전 “검토 범위” 표현 사이 충돌 존재 |

통합 운영표에서 실제로 읽힌 반복 항목은 `예식형태`, `식순체크표`, `혼인서약서`, `성혼선언문`, `축사`, `덕담`, `비고`, `음원 재생이 필요한 구간`이다. 예시 셀에는 `없음`, `당일 지참`, `X` 등으로 보이는 값이 있으나 저해상도 때문에 개별 의미를 확정하지 않는다. 아래 상태 후보는 원본 전사 결과가 아니라 이번 Product Owner가 명시한 후보값이다.

## 4. Owner 입력 → 예식장 체크표 매핑

괄호 안 출력은 **제안 형식**이다. 실제 샘플의 빈 셀을 채운 문구로 오인하면 안 된다.

| 식순 | Owner 입력 | 개별 최종 식순표 제안 | 현재 코드 상태 | 확인/위험 |
|---|---|---|---|---|
| 개식사 | 진행 여부, 현장 요청 | `개식사` + 체크사항 | 항목·활성 상태·요청 Note 존재 | 표현 규칙은 `PROPOSED` |
| 화촉점화 | 모드, 혼주 참석, 호칭, 입장·점화 순서 | `화촉점화(진행: …)` + Child 단계 | A8/B7 포함 모드와 Child 존재 | A8/B7 고정 계약 유지 |
| 신랑 입장 | 진행 여부, 입장 방식, 동선, 음악 | `신랑 입장(방법: …)` | 방식 일부 존재, 입장곡 필드 없음 | 음악은 새 필드 필요 |
| 신부 입장 | 단독/동반, 동반자, 등장 방식, 인계 지점, 음악 | `신부 입장(동반: …)` | 등장·동반·지점 일부 존재 | 관계·호칭을 추정하지 않음 |
| 신랑·신부 맞절 | 진행 여부, 요청 | `신랑·신부 맞절` | 항목·활성 상태 존재 | 낮음 |
| 혼인서약 | 진행 여부, 낭독 방식·주체, 원고 준비 상태 | `혼인서약(낭독: …)` | 방식 일부 존재; 전용 원고 상태는 불충분 | 상태 모델 보강 필요 |
| 성혼선언 | 진행 여부, 선언 주체, 관계·표시 호칭, 원고 상태 | `성혼선언(진행: …)` | `speakerMode`, `PersonRef`, 원고 상태 일부 존재 | 역할·호칭 확정 필요 |
| 덕담 | 진행자, 관계·호칭, 원고 상태 | `덕담(진행: …)` | 단일 `speech`의 `words` 모드 | 축사와 동시 진행 불가 가능성 |
| 축사 | 진행자, 관계·호칭, 원고 상태 | `축사(진행: …)` | 단일 `speech`의 `congratulatory` 모드 | 덕담과 별도 행인 원본과 `CONFLICT` |
| 축가 | 무대 수, 곡 수, 공연자, 관계, 곡명, MR·재생 시점 | `축가 N곡` + 공연 요약 | 공연 카드·공연자·관계·곡명 존재 | 카드 1개=1곡인지 `UNKNOWN` |
| 양가 부모님 및 내빈께 인사 | 양가 참석, 호칭, 포옹 생략, 인사 순서 | `양가 부모님 및 내빈께 인사` + 체크사항 | 참석·포옹 생략 일부 존재 | UI의 사용자 호칭 입력 범위 재감사 필요 |
| 행진 | 진행 여부, Cue, 음원, 현장 요청 | `행진` + 음원·Cue 요약 | 항목·Cue·Note 존재 | 음원 자산 필드 없음 |

### 4.1 현재 코드와 항목별 분류

아래 단계에서 `현재 MVP`는 변환에 사용할 원천 데이터가 기준 브랜치 코드에 있다는 뜻이다. W2.4 로컬 변경에는 Owner 최종 확인 단계의 읽기 전용 체크표 Preview가 추가됐지만, 개별 최종 식순표 완성본, 통합 운영표, Snapshot 화면·route는 구현되어 있지 않다.

| 체크표 항목 | 현재 데이터 위치 | 현재 상태 | 필요한 변환 | 새 필드 필요 | 단계 | 근거 |
|---|---|---|---|---|---|---|
| 식순명·순서 | `CeremonyItem.title/order/id` | 현재 데이터에 이미 존재 | 안정 ID 기준 정렬 | 아니요 | 현재 MVP | 모델·템플릿 |
| 진행 여부 | `CeremonyItem.active` | 현재 데이터에 이미 존재 | `false`는 삭제가 아닌 미진행으로 출력 | 아니요 | 현재 MVP | DEC-005 |
| 진행·낭독자 | `participants`, `detailConfig.speaker` | 일부만 존재 | 역할별 표시명 생성 | 경우에 따라 | 파일럿 전 최소 기능 | 모델 |
| 관계·호칭 | `PersonRef.relation/displayTitle`, 일부 가족 호칭 | 일부만 존재 | 이름과 현장 호칭 분리 | 입력 경로 보강 가능 | 파일럿 전 최소 기능 | 모델·UI 감사 |
| 입장 방법 | `mode`, `appearance`, `escort`, 지점 필드 | 일부만 존재 | 체크표용 요약 함수 | 입장곡은 필요 | 파일럿 전 최소 기능 | 모델 |
| 혼인서약 준비 상태 | 전용 상태 없음 | 새 데이터 필드 필요 | 운영 상태로 요약 | 예 | 파일럿 전 최소 기능 | 운영표 원본 |
| 성혼선언문 준비 상태 | `manuscriptStatus` 일부 | 일부만 존재 | 공통 자료 상태로 확장 | 예 | 파일럿 전 최소 기능 | 모델·운영표 원본 |
| 덕담·축사 | `speechType`, `speaker`, `manuscriptStatus` | 이름/표현 변환 필요 + 구조 충돌 | 별도 행 투영 또는 모델 결정 | 동시 진행 시 예 | 파일럿 전 최소 기능 | 원본과 Canonical 비교 |

| CRM·계약·결제 | 없음 | 파일럿 이후로 미룸 | 이 문서 범위 밖 | 예 | 파일럿 후 CRM | DEC-012 |

## 5. 예식장 체크표 → 사회자 대본 매핑

개별 최종 식순표와 MC 대본은 서로 복사하는 문서가 아니라 같은 `Ceremony Data`를 각자 투영한 결과여야 한다.

| 기준 데이터 | 개별 최종 식순표 | MC 대본 영향 | 통합 운영표 영향 |
|---|---|---|---|
| `active` | 진행/미진행 표시 | `false`면 대본에서 제외 | 생략 여부 또는 확인 상태 요약 |
| 순서·안정 ID | 현장 실행 순서 | 같은 순서로 대본 생성 | 개별 표 링크·최종 상태 추적 키 |
| 진행자·낭독자·호칭 | 식순 옆에 짧게 표시 | 소개 문장과 본문 생성 | 원고 준비 대상 판정에 사용 |
| 원고·자료 상태 | 체크사항에 표시 | 대본 본문 또는 준비 경고 | 혼인서약서·성혼선언문·축사·덕담 상태 열 |
| 공연 카드 | 유형·공연자·곡 수·곡명 요약 | 공연 소개 문장 | 음원 준비·재생 구간 요약 |
| Cue | 현장 실행 신호 | 대본 카드의 Cue | 음원 재생이 필요한 구간 열 |
| Note | 공동 확인이 필요한 현장 메모 | 사회자 전달 Note | 운영 비고 중 공유 가능한 내용만 요약 |

현재 `generateScript(draft)`는 Owner 미리보기와 MC 화면에 공통 `ScriptPackage`를 제공한다. 따라서 새 체크표가 대본 문자열을 다시 조립하기보다, 같은 Draft를 입력으로 받는 별도 순수 투영 함수를 사용하는 것이 안전하다.

제안 인터페이스 이름은 다음과 같다. 이름과 필드는 구현 승인이 아니다.

- `CeremonyData`: 현재 `CeremonyDraft`를 감싸거나 이어받는 기준 데이터 후보
- `IndividualCeremonyChecklistView`: 한 예식 최종 식순표 투영 결과
- `WeekendOperationsChecklistView`: 여러 예식의 운영 요약 투영 결과
- `FinalCeremonySnapshot`: 확정 시점의 불변 버전 후보. 지금 즉시 필수로 확정하지 않음

## 6. Cue와 Note 처리 기준

1. Cue와 Note는 같은 기준 데이터에서 생성하되 의미를 합치지 않는다.
   - Cue: 언제 무엇을 실행할지 알려 주는 짧은 신호
   - Note: 현장 요청·주의·준비사항
2. 현재 `cueOverride`는 자동 Cue를 대체하고, `requestNote`는 Note에 추가된다.
3. 개별 최종 식순표에는 식순별 핵심 Cue와 공동 확인 Note를 표시할 수 있다.
4. 통합 운영표에는 모든 대본 문장을 싣지 않고 음원 재생 구간과 해결되지 않은 운영 Note만 요약한다.
5. 개인정보·내부 메모·역할별 비공개 메모의 노출 범위는 현재 모델에 없다. 한 Note 필드를 모든 역할에 그대로 공개하면 안 된다.
6. `active=false`인 항목은 현재 Script Engine처럼 대본·Cue·Note 실행 목록에서 제외한다. 입력값 자체는 보존한다.
7. `narrationOverride`가 있으면 자동 대본보다 우선한다. Owner의 인물 정보가 바뀌어도 직접 수정 문장 속 옛 이름·호칭이 남을 수 있으므로 “자동 기준 데이터와 직접 수정 대본 불일치” 경고가 필요하다.

### Canonical 충돌 처리

- 개별 샘플은 12개 행, 현재 코드는 주례 없음 13개·주례 있음 15개다.
- 샘플에는 `예물교환`과 `폐회식`이 없고 `덕담`과 `축사`가 각각 행으로 있다.
- 현재 주례 없음 Canonical에는 `예물교환`, `폐회식`이 있으며 `덕담/축사`는 한 항목의 모드 선택이다.
- 샘플에는 주례 관련 행이 없지만 이것만으로 “주례 없음 예식”이라고 확정하지 않는다.
- 이 차이는 `CONFLICT`이며, 이번 문서에서 샘플 또는 Canonical 한쪽을 정답으로 바꾸지 않는다.

### 9.3 상태값 후보

이번 Product Owner가 제시한 후보를 다음처럼 분리해 검토한다.

| 사용자 표시 | 코드 후보 | 의미 |
|---|---|---|
| 확인 전 | `unchecked` | 아직 확인을 시작하지 않음 |
| 확인 필요 | `needs_review` | 내용·자료를 다시 확인해야 함 |
| 확인 완료 | `confirmed` | 담당자가 현재 내용을 확인함 |
| 당일 지참 | `bring_on_day` | 사전 파일 없이 당일 실물을 준비함 |
| 생략 | `omitted` | 해당 자료나 절차가 필요하지 않음 |
| 미제출 | `not_submitted` | 필요한 자료가 아직 제출되지 않음 |

`active=false`는 식순의 미진행 상태이고, 자료 상태 `omitted`는 문서·준비물의 생략 상태다. 같은 값으로 합치면 안 된다. 상태 전이, 누가 확인할 수 있는지, 확인 완료 후 변경 시 자동으로 `확인 필요`로 되돌릴지는 Product Owner 결정이 필요하다.

### 9.4 날짜·시간 정렬

통합 운영표의 기본 정렬은 `weddingDate ASC → startTime ASC → weddingId ASC`를 추천한다. 같은 시간의 홀 구분이 필요하면 `hallName`을 `weddingId` 앞에 둘 수 있으나 운영 방식 확인이 필요하다. 시간대와 timezone은 문자열 표시와 정렬 가능한 값으로 분리한다.

### 9.5 예식장·사회자 보기 권한

- 현재 localStorage MVP에는 로그인·역할 권한이 없어 기술적으로 보기를 제한할 수 없다.
- 파일럿 임시안은 지정된 한 대의 운영 기기·한 브라우저에서만 사용할 수 있다.
- 예식장과 사회자가 각 기기에서 같은 최신 표를 보려면 서버 저장, Auth, 역할, RLS 또는 동등한 권한 모델이 필요하다.
- 서버 구현 전에는 “자동 동기화”, “동시 편집”, “최신본 보장”을 완료로 표시하면 안 된다.

### 9.6 인쇄·PDF 최소 열

통합 운영표의 인쇄 후보 열은 날짜, 시간, 운영용 예식 식별명, 홀, 예식 형태, 개별 식순표 상태, 혼인서약서, 성혼선언문, 축사, 덕담, 음원 재생 구간, 비고, 마지막 수정 시간이다. 화면 링크는 종이에서 작동하지 않으므로 별도 짧은 참조 번호가 필요할 수 있다. QR은 이번 범위가 아니다.

## 10. 현재 MVP와 파일럿 이후 범위 구분

### 10.1 파일럿 전 목표 후보 — Product Owner 결정 필요

- 개별 예식 최종 식순표: 파일럿 전 최소 기능 후보
- 여러 예식 통합 운영 체크리스트: 핵심 운영 목표이지만 여러 예식 관리·권한·서버 저장과 구현 시점을 함께 결정
- 두 결과물을 구현할 경우 개별 정보 변경 시 같은 기준 데이터에서 자동 갱신
- 날짜·시간 정렬, 자료별 준비 상태, 비고, 음원 재생 구간
- 통합표에서 개별 최종 식순표로 이동
- 예식장과 사회자가 같은 최신 정보 확인: 서버 없는 임시안으로는 완전 보장 불가
- 인쇄·PDF 가능성 검토

### 10.2 현재 구조에서 가능한 최소 구현안

1. 기존 `CeremonyDraft`를 입력으로 받는 순수한 개별 체크표 투영 함수를 만든다.
2. 대본은 기존 Script Engine을 계속 사용하고, 체크표는 같은 안정 ID와 순서를 참조한다.
3. 여러 예식 저장은 기존 Draft key/schema를 즉시 바꾸지 말고 별도 Batch에서 repository·migration·rollback을 승인한다.
4. 서버 없이 시험할 때는 지정 운영 기기 한 대에 여러 예식을 저장하는 local-only repository와 목록 화면을 별도 승인해 사용할 수 있다.
5. 통합표는 저장된 각 예식에서 운영 필드만 매번 계산하여 수동 복사 없이 자동 갱신한다.
6. 인쇄는 개인정보 마스킹과 페이지 열 폭을 먼저 확정한 뒤 별도 Export Batch로 구현한다.

### 10.3 서버 저장 없는 임시 파일럿의 한계

- 다른 기기·브라우저와 자동 동기화되지 않는다.
- 예식장과 사회자가 동시에 같은 최신본을 본다고 보장할 수 없다.
- 역할별 접근 통제, 변경자 추적, 충돌 해결, 중앙 백업·복원이 없다.
- 브라우저 데이터 삭제·손상 시 복구가 어렵다.
- 수동 파일 전달을 사용하면 최신본 혼동과 개인정보 유출 위험이 생긴다.

따라서 local-only 안은 데이터 투영과 현장 표 형식을 검증하는 임시 수단일 뿐, Product Owner가 요구한 공동 최신표의 완전한 운영 해법은 아니다.

### 10.4 Final Ceremony Snapshot 판단

| 선택 | 장점 | 위험·한계 | 추천 |
|---|---|---|---|
| 현재 Draft에서 매번 투영 | 변경 즉시 체크표·대본·통합표 동기화, 기존 구조 재사용 | “확정본”이 계속 바뀌어 현장 기준 고정이 어려움 | 첫 투영 Batch에 적합 |
| Final Snapshot 즉시 구현 | 확정 시점과 버전 명확 | 권한·해제·재확정·변경 비교·migration이 함께 필요 | 이번 Batch에서 구현하지 않음 |
| Snapshot 계약 먼저 문서화 | 위험을 먼저 결정하고 구현 가능 | 실제 잠금 기능은 없음 | **추천** |

파일럿 전 “확정” 의미가 필요하므로 Snapshot 또는 동등한 버전 고정 계약은 파일럿 전에 결정해야 한다. 그러나 이번 문서만으로 Final Snapshot 구현을 필수 확정하지 않는다. 먼저 Draft 투영을 만들고, 별도 Finalization Contract Batch에서 확정·재확정 규칙을 승인하는 순서를 추천한다.

### 10.5 파일럿 검증·파일럿 후·장기

- 파일럿 전 최소 기능: 다중 기기 공동 운영을 하려면 서버 저장·역할별 읽기 권한·개인정보 정책을 별도 승인·구현. 지정된 한 대의 기기만 쓰는 임시 검증이라면 한계를 명시
- 파일럿 검증: 구현된 권한과 동기화가 실제 현장에서 최신본을 보장하는지, 인쇄 레이아웃과 변경 후 재확인 흐름이 작동하는지 확인
- 변경 이력: 중요하지만 이번 1차 범위에서는 설계 후보. 상세 구조는 별도 Batch
- 파일럿 후 CRM: 예식장 CRM, 계약·결제
- 장기 Wedding Data OS: 파트너·상품 판매, 분석·확장 플랫폼

## 14. 개발자 전달용 한 페이지 요약

### 가장 중요한 원칙

신랑·신부가 같은 내용을 두 번 쓰게 만들지 않는다. 사회자나 예식장도 내용을 복사해서 새 표를 만들지 않는다. **한 번 입력한 하나의 기준 데이터**를 역할에 맞게 다르게 보여준다.

```text
A. 한 예식
신랑·신부 입력 → Ceremony Data → 개별 예식 최종 식순표
                                  ├→ MC 대본
                                  └→ Cue / Note

B. 여러 예식
각 예식의 같은 Ceremony Data
    → 개별 최종 식순표의 운영 요약
    → 날짜·시간별 주말 통합 운영 체크리스트

통합 운영 체크리스트는 개별 표의 원문을 복사한 문서가 아니다.
같은 기준 데이터에서 준비·확인 상태만 자동 요약한 별도 결과물이다.
```

### 지금 있는 것

- 화촉점화 A8/B7
- Owner와 MC가 함께 쓰는 Script Engine
- 항목을 꺼도 값을 지우지 않는 `active=false`
- 정렬해도 바뀌지 않는 안정 ID

- 예식 시작 시간과 날짜·시간 정렬
- 개별 최종 식순표와 통합 운영표 화면
- 자료별 `확인 전/필요/완료/당일 지참/생략/미제출` 상태
- 음원 재생 구간의 정식 데이터
- 확정본 버전과 역할별 권한
- 서로 다른 기기 사이의 자동 동기화

### 먼저 해야 할 것

1. 현재 Draft에서 개별 체크표와 운영 요약을 만드는 변환 규칙을 고정한다.
2. 실제 샘플과 Canonical의 12 대 13/15 충돌, 덕담·축사 구조를 Product Owner가 결정한다.
3. 여러 예식 저장을 만들기 전에 기존 Draft를 잃지 않는 migration·rollback을 설계한다.
4. 통합표는 저장된 예식들을 날짜와 시간으로 정렬하고, 각 예식의 상태를 자동 계산한다.
5. 다른 기기에서도 같은 최신표를 보여 주려면 서버·로그인·권한을 별도 Batch로 만든다.

이 문서의 데이터 정책은 여전히 Product Owner가 검토할 `PROPOSED` 제안이다. 위 W2.4 로컬 구현 상태만 코드와 자동 테스트로 확인했으며 Final Snapshot·통합 운영표 완료를 의미하지 않는다.
